🎯 My Journey with Android System Programming

Most Android developers stay comfortable in the app layer - Activities, Fragments, ViewModels. But I decided to dive deeper into Android's system architecture. That's how my AOSP HAL Demo project was born - an educational exploration of JNI integration and hardware abstraction layer concepts.

This project became my learning playground for understanding two key areas: LED control and haptic feedback, but from a system programming perspective:

🔦 FlashTune - Pure JNI/HAL

My educational LED controller that demonstrates authentic AOSP development patterns. No Camera2 APIs - just pure JNI to C++ HAL implementation. It's designed for learning how Android system programming really works, with direct sysfs interface access where available.

📳 PulsePlay - Hybrid Approach

Learning hardware abstraction principles:

// led_hal.cpp - Educational HAL implementation
void led_hal_turn_on() {
    const char* led_path = "/sys/class/leds/led0/brightness";
    if (access(led_path, W_OK) == 0) {
        std::ofstream led_file(led_path);
        if (led_file.is_open()) {
            led_file << "255";  // Educational sysfs write
            led_file.close();
        }
    } else {
        // Educational logging for devices without hardware
        __android_log_print(ANDROID_LOG_INFO, "LED_HAL", "LED turned on (simulated)");
    }
}

The file browser needed to handle different file types gracefully and provide smooth navigation:

// BrowseActivity.java - Directory listing with file type detection
private void loadRemoteDirectory(String path) {
    new Thread(() -> {
        try {
            ChannelSftp sftpChannel = (ChannelSftp) session.openChannel("sftp");
            sftpChannel.connect();
            
            Vector<ChannelSftp.LsEntry> files = sftpChannel.ls(path);
            List<ChannelSftp.LsEntry> sortedFiles = new ArrayList<>();
            
            // Smart sorting: directories first, then files alphabetically
            for (ChannelSftp.LsEntry entry : files) {
                if (!entry.getFilename().startsWith(".")) {
                    sortedFiles.add(entry);
                }
            }
            
            runOnUiThread(() -> {
                adapter.setFiles(files);
            });
        } catch (Exception e) {
            runOnUiThread(() -> showError(e.getMessage()));
        }
    }).start();
}

// File transfer with progress monitoring
private void downloadFile(String remotePath, String fileName) {
    ProgressDialog progressDialog = new ProgressDialog(this);
    progressDialog.setMessage("Downloading " + fileName);
    progressDialog.show();
    
    new Thread(() -> {
        try {
            File localFile = new File(getExternalFilesDir(null), fileName);
            FileOutputStream fos = new FileOutputStream(localFile);
            
            sftpChannel.get(remotePath, fos);
            
            runOnUiThread(() -> {
                progressDialog.dismiss();
                Toast.makeText(this, "Downloaded: " + fileName, Toast.LENGTH_SHORT).show();
            });
        } catch (Exception e) {
            runOnUiThread(() -> showError(e.getMessage()));
        }
    }).start();
}

🔐 Multiple Encryption Algorithms

CipherGuard supports four industry-standard encryption methods, each optimized for different use cases:

🥇 AES-GCM (256-bit) - Recommended

Authenticated encryption with built-in integrity verification. Galois/Counter Mode provides both confidentiality and authentication in one operation. Perfect for most use cases.

🔒 AES-CBC (256-bit) - Traditional

Cipher Block Chaining with PKCS7 padding. The traditional AES mode for compatibility with legacy systems. Reliable and widely supported.

🔑 RSA-OAEP (2048-bit) - Hybrid

Hybrid encryption using RSA for key encryption and AES-GCM for data. Optimal Asymmetric Encryption Padding with SHA-256. Ideal for large files.

🗝️ AES-PBKDF2 - Password-Based

Password-based encryption with 100,000 iterations using PBKDF2-HMAC-SHA256. Secure key derivation with random salts for personal data protection.

🔐 Android KeyStore Integration

Hardware-backed security using Android's Trusted Execution Environment:

// CryptoUtils.java - Secure key generation
private static SecretKey getOrCreateAESKey(String alias, String blockMode) throws Exception {
    KeyStore keyStore = KeyStore.getInstance("AndroidKeyStore");
    keyStore.load(null);
    
    if (!keyStore.containsAlias(alias)) {
        KeyGenerator keyGenerator = KeyGenerator.getInstance(
            KeyProperties.KEY_ALGORITHM_AES, "AndroidKeyStore");
            
        KeyGenParameterSpec.Builder builder = new KeyGenParameterSpec.Builder(alias,
            KeyProperties.PURPOSE_ENCRYPT | KeyProperties.PURPOSE_DECRYPT)
            .setBlockModes(blockMode)
            .setKeySize(256)  // 256-bit AES
            .setEncryptionPaddings(blockMode.equals(KeyProperties.BLOCK_MODE_GCM) 
                ? KeyProperties.ENCRYPTION_PADDING_NONE 
                : KeyProperties.ENCRYPTION_PADDING_PKCS7);
                
        keyGenerator.init(builder.build());
        keyGenerator.generateKey();
    }
    
    return (SecretKey) keyStore.getKey(alias, null);
}
🎯 AES-GCM Implementation

Authenticated encryption with proper IV handling:

// AES-GCM encryption with authentication
public static String encryptAESGCM(Context context, String plainText) throws Exception {
    SecretKey key = getOrCreateAESKey(AES_GCM_ALIAS, KeyProperties.BLOCK_MODE_GCM);
    
    Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
    cipher.init(Cipher.ENCRYPT_MODE, key);
    
    byte[] iv = cipher.getIV();  // 12-byte IV for GCM
    byte[] encrypted = cipher.doFinal(plainText.getBytes(StandardCharsets.UTF_8));
    
    // Combine IV + encrypted data
    byte[] combined = new byte[iv.length + encrypted.length];
    System.arraycopy(iv, 0, combined, 0, iv.length);
    System.arraycopy(encrypted, 0, combined, iv.length, encrypted.length);
    
    return Base64.encodeToString(combined, Base64.DEFAULT);
}
